# GRPO Integration Bugs — 2026-03-03

## Summary

rl-values uses Tome as the inference server for GRPO training. Training produced diverging log probs (10^15 KL divergence). Root cause was multiple bugs in Tome's rollout and weight update paths.

## Bugs Fixed

### 1. Decode cache missing prefill KV (CRITICAL)

`_process_rollout_chunk` and `_process_ref_logprobs_batched` created a fresh `batched_cache` for the G-way decode phase but never populated it with the prefill KV data. The model decoded with empty context — logits were garbage.

**Fix** (`node.py`): After prefill, `mx.repeat` the prefill cache's `_keys`/`_values` into the batched decode cache. Pad to max prompt length when batching multiple prompts with different lengths.

### 2. Prefix cache returns uninitialized data (CRITICAL)

`prefix_cache.insert()` stored block table references after prefill, but `flush_to_pool()` was never called. The paged KV pool blocks had no data. On cache hit, `gather_kv()` read zeros/garbage.

**Status**: Disabled for now (both policy and reference prefix caches). See future work.

### 3. Weight update: missing LoRA scale

`UpdateWeights` computed `delta = B @ A` without the `_lora_scale` factor (e.g., 2.5 for rank=8, scale=20). The actual LoRA effect is `scale * B @ A`.

**Fix**: Added `float lora_scale = 7` to `LayerUpdate` in `inference.proto`. Scheduler passes it through. Node applies `scale * B @ A`.

### 4. Weight update: cumulative instead of replacement

`UpdateWeights` did `W += delta`, accumulating all historical deltas. Should be `W = base_W + current_delta`.

**Fix**: Use `self.reference_model` weights as the immutable base. Each update sets `policy_W = ref_W + scale * B @ A`.

### 5. gRPC message size limit

Default 4MB gRPC limit rejected LoRA weight updates (~7MB for 84 adapters).

**Fix**: Added `grpc.max_receive_message_length = 64MB` option to the gRPC server.

### 6. HTTP body size limit

Axum default 2MB body limit rejected the `/v1/weights` POST.

**Fix**: Added `DefaultBodyLimit::max(64MB)` on the `/v1/weights` route.

## Future Work

### P0 — Fix prefix cache

The prefix cache is the key performance optimization for GRPO (all G rollouts share the same prompt prefix). Currently disabled.

**What needs to happen:**
1. After prefill, call `cache.flush_to_pool()` to write KV data to the paged block pool
2. On cache hit, call `cache.gather_kv(l)` to restore KV from pool into contiguous cache
3. The code for this exists but the `update_paged_kv` Metal kernel uses an in-place write trick (`device T* k_pool_mut = (device T*)k_pool`) that may not work reliably with MLX's graph-based evaluation — the "input" array is mutated as a side effect, but MLX doesn't track this mutation

**Options:**
- **A**: Rewrite `update_paged_kv` to return the modified pool as an output (not in-place). This is the cleanest fix but means the pool arrays get replaced each flush.
- **B**: Use `mx.eval()` barriers carefully to ensure the kernel's side effects are visible before any subsequent read. Current code does `mx.eval(self.allocator.k_pool + self.allocator.v_pool)` after flush, but this may be a no-op since the arrays were already materialized.
- **C**: Skip the Metal kernel entirely and do the scatter in Python (slower but correct). Good for testing.

**Test**: Run the consecutive-rollout test (`test_logprob_comparison.py`) — cold and warm cache should produce same first-token distribution.

### P1 — Sampling parity

Tome's `_sample_token` uses top-k=20 sampling. rl-values' local `sample_group` uses full categorical sampling (`mx.random.categorical(logits/temp)`). These should use the same strategy for consistent behavior. Either:
- Make top_k configurable in the rollout request
- Or remove top_k from Tome (use full categorical like the local path)

### P1 — Error propagation for weight updates

`update_weights` silently returns errors as dict values. The training script doesn't check. A failed weight sync means Tome's policy diverges from the trainer's, causing exactly the KL explosion we debugged. Should raise on failure.

### P2 — Shared model code

Tome and rl-values have diverged copies of `model.py` / `kvcache.py` / `load_weights.py`. Both produce identical outputs (verified). Consider symlinking or making a shared package to prevent future drift.

### P2 — bf16 vs f32 log prob precision

Tome's `fused_log_softmax` returns bf16 (max ~0.75 diff from f32). rl-values computes log probs in f32 during the gradient pass. The gap (~0.44 per token) is within training noise but could be tightened by casting to f32 before extracting the scalar log prob in Tome.
